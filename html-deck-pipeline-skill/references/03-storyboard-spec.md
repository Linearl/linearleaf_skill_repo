# 分镜稿规范（通用）

> 迁移来源：`.github/skills/html-deck-pipeline-skill/references/v1-storyboard-spec.md`
>
> 适用范围：`html-deck-pipeline-skill` 的全部主题、全部风格。

## 1. 文件级规范

### 1.1 Frontmatter（必填）

分镜稿顶部使用 YAML 头，建议包含：

- `version`
- `updated`
- `source_html`
- `total_pages`
- `owner`

### 1.2 标题与来源说明（必填）

Frontmatter 之后建议固定包含：

1. 一级标题（示例：`# 02-00-01 开头分镜稿`）
2. 来源与覆盖范围说明（`> 来源` / `> 覆盖`）
3. 分隔线 `---`

---

## 2. 页面级规范（核心骨架）

每一页使用统一块结构：

1. 二级标题：`## 第X页｜页面标题`
2. `### 页面目标`
3. `### 页面完整文案`
4. `### 演讲备注`

### 2.1 标题格式

- 必须使用“第X页 + 竖线 + 标题”格式：`## 第X页｜...`
- 每个分镜文件的第 1 页建议为导航页（除封面专用分镜外）。

### 2.2 页面完整文案中的元字段

建议稳定使用如下字段：

- `页眉主题：...`
- `标题：...`
- 视场景补充：`标签：...`、`补充：...`

> 建议：避免混用 `页内主题`，统一使用 `页眉主题`。

-### 2.2.1 Emoji/符号美化要求（新增）

> **分工说明**：本节为 Emoji/符号的"设计与写作指南"（元素库、映射表、覆盖建议）；机器可检查的门禁规则（正则、ERROR 判定）见 `07-quality-gate-patterns.md` §1.4。两者互补不重复。

- 阶段 C 产出分镜时，必须直接完成 Emoji/符号美化并冻结，不得延后到 HTML 阶段再补。
- 建议参考历史风格：`11-组内分享/docs/3.其他/v1`（仅参考呈现风格，不复制具体内容）。
- 分镜稿在描述标题眉标、标签、步骤图标、提示框时，应显式规划 Emoji/符号呈现，不要默认退回纯 ASCII 文案。
- Emoji/符号的目标是**强化语义与层级**，不是堆砌装饰。应先写清信息结构，再补符号；若去掉符号后文案不可读，说明设计方向已偏。
- 推荐优先级应遵循：
   1. **语义型 emoji 优先**：`✅`、`❌`、`⚠️`、`💡`、`📌`、`🎯`
   2. **流程连接符号其次**：`→`、`➜`、`➡️`
   3. **结构型符号按需**：`◉`、`✦`、`◆`、`◇`
- 若页面是流程页、行动页、提示页、对比页，分镜稿中应提前写明对应的 Emoji/符号呈现方式，供 HTML 模板直接落地。
- 流程步骤优先使用语义 emoji（如 `🎯/🔍/🛠️/🧪/✅`），不强制使用 emoji 数字序号（如 `1️⃣/2️⃣/3️⃣`）。

建议将本节理解为“符号设计约束”，而不是“装饰备忘录”。参考现有 deck 实际风格，可优先复用 `🧭`、`🎯`、`✅`、`❌`、`→`、`⚠️`、`💡`、`🚀`、`📌` 等稳定元素。

（1）Emoji/符号元素库（按用途分组）

| 类别 | 主推荐 | 备用 | 典型语义 | 何时不用 |
| --- | --- | --- | --- | --- |
| 眉标 / 页签 | `◉`、`◆`、`◇` | `📍`、`🎯` | 章节眉标、页签、当前节点、主题起始 | 同一页已经有明显色块页签时，不必再叠加多个眉标 |
| 标签 / Chip | `✦`、`✳`、`▣` | `🏷️`、`📎` | 短标签、主题标签、方法标签、版本标签 | 标签本身已很长，或标签数量过多时，避免每个都挂符号 |
| 对比 / 决策 | `✓`、`✗`、`→`、`VS` | `✅`、`×`、`❌` | 正反例、优劣判断、推荐路径、旧版/新版选择 | 同一对比块内不要同时出现 `✓` 和 `✅`，也不要混用 `✗` 和 `×` |
| 流程 / 步骤 / 箭头 | `🎯`、`🔍`、`🛠️`、`🧪`、`✅`、`→`、`⇒` | `① ② ③`、`STEP 1/2/3` | 流程推进、阶段拆分、顺序执行、收敛方向 | 表格已自带序号列、代码块、命令行、路径名中，优先保留纯文本 |
| 状态 / 风险 / 提示 | `ℹ`、`⚠`、`✅`、`📌`、`💡` | `🔑`、`❗` | 提示框、风险框、验收项、行动提醒、关键结论 | 不要把每一条普通 bullet 都升级成提示符号 |
| 场景图标（emoji） | `🚀`、`🎯`、`📍`、`🧠`、`⚙️`、`👥`、`📅`、`⏱️` | `📚`、`🔧`、`🧭` | 封面、章节场景、元信息、角色/对象、时间/节奏 | 证据表、密集数据区、严肃结论区应少用 emoji，避免干扰扫描 |

（2）符号映射速查表（推荐直接在分镜稿沿用）

| 角色 | 推荐令牌 | 说明 |
| --- | --- | --- |
| 章节眉标 | `◉` | 默认最稳健，适合标题眉标、页签、章节起始 |
| 标签 / Chip | `✦` | 适合短标签、主题标签、方法标签 |
| 正向判断 | `✓` | 适合对比页、清单页、决策结论 |
| 反向判断 | `✗` | 适合反例、禁忌、错误做法 |
| 步骤编号 | `①` | 流程页优先，能直接体现顺序 |
| 流程连接 | `→` | 线性流程默认箭头 |
| 风险提示 | `⚠` | 风险、限制、失败分支 |
| 说明提示 | `💡` | 解释、启发、阅读提示 |
| 行动提醒 | `📌` | 待执行动作、落地要求、记住事项 |
| 收口 / 发射 | `🚀`、`🎯` | 收口页、目标页、行动号召页 |

如需交给他人复用，建议在分镜首页或章节首页显式写出一组风格令牌，例如：

```text
主符号体系：🧭 / 🎯 / ✅ / ❌ / 🔍 / 🛠️ / 🧪 / → / ⚠️ / 💡 / 📌
替补符号体系：📍 / 🏷️ / ✔️ / ✖️ / ① / ➜ / 🚨 / ℹ️ / 🔁
```

（3）统一性约束（同一 deck 必须遵守）

> 机器检查版本见 `07` §1.4（`ERROR` 判定规则）。以下为设计层面的完整约束。

1. **先选主符号体系，再写页面。** 推荐每个 deck 固定一套主符号体系，并预留一套替补体系。
2. **同一语义只对应一个主符号。** 例如"正向判断"统一用 `✓` 或 `✅`（违反此条在 `07` §1.4 中会判 `ERROR`）。
3. **同一结构只对应一种步骤方案。** 不建议跨页混用多套编号规则（违反此条在 `07` §1.4 中同样判 `ERROR`）。
4. **emoji 仅作场景增强，不替代结构骨架。**
5. **禁用混搭示例**见 `07` §1.4 不通过示例。

（4）按页面类型的最小覆盖建议

- **导航页 / 章节页**：至少覆盖 `1` 个眉标 / 页签符号 + `1` 个路线或提示符号；若有“观看建议”，可再补 `💡` 或 `ℹ`。
- **对比页**：至少覆盖 `1` 组固定判断符号（`✓/✗` 或 `✅/×`）+ `1` 个决策/收敛符号（如 `→`、`VS`）。
- **流程页**：至少覆盖 `1` 套固定步骤体系（如 `① ② ③`）+ `1` 套固定箭头体系（如 `→` 或 `⇒`）。
- **行动页**：至少覆盖 `1` 个行动提醒符号（`📌`）+ `1` 个验收/完成符号（`✅`）；若存在风险前提，再补 `⚠`。
- **收口页 / 总结页**：至少覆盖 `1` 个结论或目标符号（`🎯`、`📌`）+ `1` 个收口或发射符号（`🚀`、`✅`）；避免堆满 emoji 墙。

（5）何时不要强行使用 Emoji/符号

- 代码块、命令行、路径名、API 字段名、配置键名等需要精确复制的内容。
- 高密度表格、证据表、数据表中，符号会破坏对齐或妨碍扫描时。
- 同一块信息已经通过颜色、位置、边框充分表达层级时。
- 符号成为唯一信息来源时。例如只写 `⚠` 而不写“风险点：...”，不符合语义优先原则。
- 目标展示环境字体不稳定、存在 tofu 方块或宽度异常风险时，应回退到替补体系或纯文本。

（6）人工可校验检查项（不要求写脚本）

1. 抽查任意 3 页，确认同一语义是否始终使用同一主符号。
2. 检查是否已记录主符号体系与替补体系，且替换关系可说清。
3. 检查页面标题区、主体区、提示区是否存在“纯装饰堆砌”；若删除符号后信息失真，说明设计过度依赖装饰。
4. 检查对比页、流程页、行动页是否满足对应的“最小覆盖建议”，避免该有的符号缺席。
5. 检查高密度表格、代码块、命令行是否仍保持纯文本可复制，不被 Emoji/符号干扰。
6. 检查 emoji 是否只用于场景增强，而不是充当默认 bullet。
7. 在常见系统字体下人工预览一次，确认没有乱码、空框、过宽导致的换行异常。

### 2.3 演讲备注中的转场要求

每页演讲备注建议至少包含：

- 讲述重点（先结论再证据）
- 过渡句（显式写 `过渡句：...`）

### 2.4 页面文案量与信息密度（新增）

为避免“分镜稿过于简略、页面内容过少”，页面完整文案建议满足以下口径：

- **每页建议至少 2~4 个信息块**（如路线图、卡片组、对比块、步骤流、表格、提示框）。
- 对比页建议至少 **3~4 组对比项**，不得只给 1 组示例。
- 方法页/流程页建议同时包含：结论型标题、结构化主体（步骤流/表格/矩阵之一）、边界说明或失败分支提示。
- 行动页建议同时包含：触发条件、关键动作、验收信号；推荐补充口播版三步法。
- 导航页与总结页均应保留，避免章节只剩“方法页堆叠”。

> 目标不是无上限加字数，而是确保每页“信息足够支撑独立讲解与独立成页”。

## 3. 内容组织模式（通用）

### 3.1 导航页模式

用于章节起始页，通常包含：

- 路线图（本章子主题顺序）
- 观看建议
- 提示框（如何理解本章）

### 3.2 方法页模式

用于讲单个方法，常见组合：

- 结论型标题
- 对比块（反例/正例）
- 步骤流程（1,2,3,4）
- 模板或表格（便于即用）

### 3.3 行动页模式

用于章节收口，建议统一组织为：

- 触发条件
- 关键动作
- 验收信号

### 3.4 页面元素库（用于约束后续 HTML 生成）

为提升后续 HTML 可控性，分镜中建议显式声明页面元素模块。以下元素可作为“通用元素库”：

- **路线图/路径图**：用于导航页或章节起始页（建议至少 3 节点）
- **卡片组（Card Group）**：用于要点、方法、行动拆解（建议至少 2 张）
- **提示框（Tip Box）**：用于风险提醒、阅读提示、执行边界（建议每个核心章节至少 1 处）
- **对比块（Compare Block）**：用于正反例、旧版/新版、方案 A/B（对比页必备）
- **步骤流（Step Flow）**：用于方法/流程页（建议 3 步及以上）
- **证据表或矩阵（Table/Matrix）**：用于证据页（需可复核字段）

> 推荐：在元素标题或字段值中直接写出 Emoji/符号装饰示例，例如 `#### 🎯 标签`、`#### 🔍 步骤流`、`#### 💡 提示框`，便于后续 HTML 模板稳定继承。

> 建议在 `### 页面完整文案` 中直接写明元素标题，例如：`#### 路线图`、`#### 卡片`、`#### 提示框`、`#### 对比`，便于脚本或人工审校。

## 4. 条件触发的增强规范

以下条款建议作为“条件必加”规则：

1. **行动页**：包含 `口播版三步法`
2. **命令/流程/高风险页**：包含 `失败分支提示`
3. **长表格页**：包含“三条带走结论”
4. **高概念页**：包含“反例一句话”

---

## 5. 附录规范（文件结尾）

每份分镜稿建议固定包含两个附录：

1. `## 附录A：页码映射表（分镜页 -> HTML data-index）`
   - 行数与本文件页数一致
   - 页码连续、映射连续
2. `## 附录B：分镜质量检查清单`
   - 覆盖结构一致性、导航页、过渡句、行动页、映射完整性

## 7. 与门禁脚本的关系

本文件是“写作与审校规范”，用于提升分镜一致性；
`07-quality-gate-patterns.md` §1 是"机器检查门禁"，用于自动校验。

两者的边界：
- **规范冲突时**：以 `07` §1 门禁阈值为准。
- **Frontmatter 字段**：本文件 §1.1 为推荐字段，`07` §1.3 为强制门禁字段，两者并集为最终标准。
- **Emoji 规则**：本文件 §2.2.1 为写作参考，`07` §1.4 为底线检查。

若需要升级为强制门禁，可在校验脚本中添加对应规则。
