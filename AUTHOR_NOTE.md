# 作者说明：方法论还是 Workflow，形态升级为 Skill

## 0. 引言

这次从 workflow 迁移到 skill，我们做的并不是“发明一套全新逻辑”，而是把人类社会里长期验证有效的协作实践，迁移到 AI 协作系统里。

核心思路很朴素：

- 人类社会怎么在复杂分工里保持稳定协作；
- AI 系统就怎么在多智能体场景里建立可执行的组织结构。

下面是几个最直观的映射对比：

| 人类社会实践 | AI 协作映射 | 解决的问题 |
| --- | --- | --- |
| 岗位分工 + 团队协作 | sub-agent / agent-team 分工协作 | 单体智能体覆盖有限、复杂任务并行效率低 |
| 项目文档与档案体系 | workspace + subspace + 归档机制 | 过程资产散落、跨阶段信息断裂 |
| 标准流程（SOP）与质检点 | workflow 阶段推进 + 检查点 + 验收门槛 | 执行漂移、质量不稳定 |
| 会议纪要与制度化交接 | 以文档代通信 + 标准交付接口 | 协作通信带宽高、重复解释成本高 |

所以这份文档的立场很明确：**我们把人类成熟的组织与流程经验，翻译成 AI 可执行的工程机制。**

## 1. 为什么迁移到 Skill（以及本仓库定位）

我们之前做的是 **workflow 项目**，现在迁移为 **skill 项目**。这不是“推翻重来”，而是把同一套方法论换成更标准化、自动化的承载形态。

先把定位讲透：

- **Workflow 是方法论内核**：负责把复杂任务做稳、做透、可复盘；
- **Skill 是标准化外壳**：负责让能力被系统动态加载、自动接入、平滑调用。

迁移到 Skill 的直接原因也很现实：它已经成为标准，并且支持动态加载，很多场景不需要像传统 workflow 那样手动唤醒。

但方法论没有变——执行质量、过程留痕、资产沉淀这三件事，依然靠 Workflow 思维来保证。

`linearleaf_skill_repo` 的定位就是：

- 在 **Skill 标准**下组织能力；
- 用 **Workflow 机制**保证质量；
- 通过模板、目录和复盘持续沉淀过程资产。

说得口语一点：

**我们换的是接口层，不是做事的方法。**

### 术语约定（统一口径）

- **动态加载**：根据上下文自动启用能力；
- **手动唤醒**：需要显式要求进入某套流程；
- **任务工作空间（workspace）**：承载团队级任务资产的主空间；
- **智能体子空间（subspace）**：每个智能体独立记录输入/过程/输出的子空间；
- **以文档代通信**：用结构化文档传递上下游信息，减少高频对话同步。

## 2. Skill vs Workflow：优势与劣势对比

| 维度 | 项 | Skill | Workflow |
| --- | --- | --- | --- |
| 启动与接入 | 优 | 标准化程度高，动态加载顺滑，通常不必手动唤醒。 | 入口明确、流程可控，适合复杂任务。 |
| 启动与接入 | 劣 | 复杂任务里流程约束偏弱，容易“会做但不稳”。 | 常需显式进入流程，初始操作更重。 |
| 执行约束 | 优 | 轻量灵活、响应快。 | 阶段、检查点、退出条件清晰。 |
| 执行约束 | 劣 | 阶段边界和质量门槛不天然强约束。 | 对简单任务可能显得“流程偏重”。 |
| 工作空间与过程资产 | 优 | 可快速调用能力点。 | 通常维护 workspace，支持资产存储/更新/流转/归档。 |
| 工作空间与过程资产 | 劣 | 缺少统一任务工作空间，过程资产易散落。 | 需要维护目录、模板和归档规范。 |
| 复盘与传承 | 优 | 适合即时解决单点问题。 | 复盘链路完整，经验沉淀效率高。 |
| 复盘与传承 | 劣 | 跨会话复盘和经验复用成本更高。 | 前期设计模板与结构需要投入。 |
| 适用场景 | 优 | 高频、小步、上下文清晰任务表现好。 | 多阶段、多人协作、需留痕审计任务更稳。 |
| 适用场景 | 劣 | 超长链路任务易出现上下文漂移。 | 不适合所有轻量场景“一刀切”。 |

这份对比不是让你二选一，而是帮助组合使用：

> **用 Skill 提高接入效率，用 Workflow 保障执行质量与资产沉淀。**

## 3. 二者差异与多智能体协作下的 Workspace 需求

除了“标准化封装 vs 过程化约束”，更关键的差异在于：

> **Workflow 天然要求一个可持续维护的任务工作空间（workspace）。**

这个 workspace 解决的是连续性问题：

- 资产能存：计划、轮次记录、报告、复盘都能落地；
- 资产能更：每轮推进都有固定位置更新；
- 资产能流：上游结论可标准化流转到下游执行；
- 资产能复用：任务结束后可归档，下一次直接复用。

当进入 sub-agent / agent-team 协作时，这个需求会从“重要”变成“必要”：

- 团队需要**主 workspace**承载共享目标、规则、阶段计划和汇总结论；
- 每个智能体需要**自己的 subspace**记录本体输入、过程和输出；
- 主空间与子空间之间要有明确交付接口，避免口头同步与重复解释；
- 通过“以文档代通信”，显著降低协作过程中的通信带宽需求。

### 最小工作空间示意（建议基线）

```text
docs/
└── topic_x/
    ├── 00_shared/                  # 共享目标、规则、计划、汇总
    ├── 01_agent_a/                 # Agent A 子空间
    │   ├── input.md
    │   ├── worklog.md
    │   └── output.md
    ├── 02_agent_b/                 # Agent B 子空间
    │   ├── input.md
    │   ├── worklog.md
    │   └── output.md
    └── 99_merge/                   # 合并结论与最终交付
```

本质上，这就是在建立一套轻量组织架构：

> **角色分工 + 子空间隔离 + 标准化交付接口。**

没有这层结构，多智能体协作很容易变成“大家都在干活，但彼此信息不同步”。

## 4. 一句话总结

**Skill 负责“更容易被自动调用”，Workflow 负责“更可靠地完成复杂任务”；两者叠加，才是可规模化的 AI 协作方式。**
